## Overview

This plan executes the **presentation-layer redesign** of SpecManager's own board UI and document panel to the "Obsidian Flux" design system. It is a pure re-skin: no `@specmanager/core`, MCP-tool, board-server API/WS, or gate-semantics changes (confirmed against Architecture §1, §3, §4 and PRD Non-goals). All work lands under `plugins/specmanager/ui/src/` plus a new token + font layer; the `tsc -p && vite build` → `ui/dist` pipeline and the Fastify `@fastify/static` serving path are unchanged.

The work is organised into **three phases**. **Phase A** (shipped) laid the foundation — token layer, fonts, every component re-skinned, BUILD relocated, responsive at three breakpoints. **Phase B** (shipped) was a **design-fidelity pass**: a visual diff of the delivered board against the `board.png` / `panel.png` mockups surfaced that Phase A landed the palette/tokens/layout but missed the *depth* the design system calls for (tonal elevation + glass + micro-interactions) plus a few high-signal details (display type on the header title, the gradient primary CTA, and the focused right-side drawer treatment). Phase B closed exactly those gaps. **Phase C** is a **layout-correctness pass**: a live DOM measurement at 1280px against the mockups found that Phase B added depth/gradient/drawer *styling* but the underlying box model never fit the grid — populated stage cards overflow their grid row and overlap the next feature row, cards in a row are ragged (different heights, content-hugging widths), the Walkthroughs roll-up label clips its card border, and the panel header still stacks vertically instead of being a single horizontal bar. Phase C fixes the geometry only. It is still presentation-only and still gated on the same build + manual-acceptance discipline; it introduces no new product capability, data field, or endpoint, and is mostly box-model/flex/grid (likely no new colors).

## Risk & sequencing notes

- **Foundation must land first.** Tasks 1→2→3 are strictly ordered: fonts and tokens have to exist before any component task can reference `--font-*` or `--surface-*`. Everything from task 4 onward depends on task 3 (the import wiring + base reset).
- **Pinned emerald is load-bearing (Architecture §9 / Q1).** Obsidian Flux's DESIGN.md frontmatter has **no** success/done color — only `tertiary` (amber) and `error` (rose). The board needs a distinct approved/done color, so task 2 introduces a single derived token `--success: #10b981` (the exact hex confirmed in the approved architecture). Both BuildPanel done segments/pills (task 8) and the approved badge (task 6) must consume this token, not a one-off hex — otherwise the legacy-hex grep gate in task 12 fails.
- **BUILD relocation is the only structural change (task 5).** It is option A (Architecture Q3): the right-hand region is a read-only progress glance; the full `BuildPanel` overlay behavior is preserved. If the region grows complex it MAY be extracted to `ui/src/BuildRail.tsx`, but that is not assumed and not a separate task. Rollback is contained to `App.tsx` grid layout — revert `COLUMNS` to include `"build"` to restore the old column.
- **Behavior-preservation hot spots.** The optimistic-concurrency banners (409 conflict / error / saved) and the sandboxed design-brief iframe are the things most likely to be silently broken by a color-only pass. Task 7 explicitly verifies the banners and protects the iframe; task 12 re-verifies end-to-end.
- **CodeMirror can drift from the rest of the UI (task 10).** The editor theme is set inline in JS, so it must read the same CSS custom properties as the stylesheet or selection/active-line colors will diverge on True Dark.
- **What blocks what (Phase A):** 1→2→3 → {4, 6} → 5 (relocation depends on the re-skinned header/cards being in place) → 7 (panel) → {8, 9} → 10 → 11 (responsive depends on the final desktop layout existing) → 12 (verification last).
- **Phase B is additive and low-blast-radius.** It never touches `@specmanager/core`, the MCP tools, the board server, or gate semantics — same envelope as Phase A. Each change is a token addition or a class/value swap on the existing Phase-A CSS, so rollback is a per-task revert of the relevant `tokens.css` / `styles.css` / `*.tsx` hunk.
- **What blocks what (Phase B):** B1 (depth/elevation tokens) → B2 (card depth + display type) and B5 (drawer glass edge), since both consume the new tokens; B3 (gradient token + `.btn--primary` rewrite) → B4 (DocPanel re-points Approve/Save to the new classes); B6 (breadcrumb + verification) runs last after every other Phase B change is in place.
