Markdown viewer PRD

Problem

The board's document panel currently splits markdown editing into two distinct surfaces shown side by side: a raw CodeMirror source editor on the left and a marked-rendered preview on the right, toggled by a “Preview” checkbox.

This split creates real friction for the one person who lives in these documents:

  • Two mental models for one document. The author reads finished prose in the preview but edits raw ##, -, **bold** syntax in the monospace pane.
  • Half the panel is “wasted” on source. With Preview on, neither pane reaches a comfortable reading width.
  • Tables render poorly — weak borders, no header styling, cramped padding.

Goals

Replace the split editor with one WYSIWYG surface, give rendered prose a comfortable reading width (~65–80 characters per line), and ship a formatting toolbar with bold / italic / heading at minimum.

The body renders immediately as styled prose in a centered ~72ch column (max-width: 72ch; margin-inline: auto), so lines land in the 65–80 char target the PRD asks for. There is no “Preview” checkbox and no monospace pane — the rendered surface is the surface.

Builder note: this replaces the marked.parse preview path for markdown stages. The wider .panel (min(64rem,68vw)) plus the centered column is what produces the readable measure — the panel is wider but the text column is not.

Screen 2 · Edit mode — toolbar resting state

One surface to read and write; the toolbar lives with it

A draft opens editable. The caret sits in the prose (end of the Summary paragraph) and what the author edits is already rendered — a heading stays a heading, bold is bold. The old .panel__toolbar row that held the Preview checkbox is gone; in its place is the formatting bar (MarkdownToolbar.tsx). The Chat toggle survives and moves to the right of the bar.

  • v1 toolbar set (expanded per PRD answers): Bold, Italic, Heading — plus Bullet list, Link, Table, Code block. Heading carries a ▾ caret hinting at the H1/H2/H3 menu.
  • Resting style: ghost buttons (transparent, --on-surface-variant glyphs) that gain an --outline-variant border + --surface-container-high fill on hover — echoing the existing .btn--ghost language. The bar sits on --surface-container.
  • Both input paths agree: clicking a button and typing markdown (**bold**, # , - ) both mutate the same ProseMirror state via Milkdown, so they serialize to one identical markdown string (PRD goal #4).

Builder note: buttons dispatch ToolbarAction commands ("bold" | "italic" | "heading" | "bulletList" | "link" | "table" | "codeBlock") against the current selection. The whole bar is hidden/disabled when readOnly (Screen 1).

Screen 3 · Toolbar active / selected state

Selection drives the toolbar's active state

The author has selected the words “formatting toolbar” and applied Bold. Two things change versus Screen 2:

  • The Bold button reads active — --primary glyph, a color-mix(--primary 14%) fill, and a primary-tinted border (aria-pressed="true"). This is the required “active/selected” state: any mark covering the current selection lights its button, mirroring VS Code / Notion toolbars.

Builder note: the toolbar reads active marks from the ProseMirror selection on every transaction; toggling Bold a second time removes the mark and the button returns to resting. onChange emits the serialized markdown so the dirty check and Save flow are unchanged.

Screen 4 · Reading width — before / after

Wider panel, narrower text

The width fix is two moves working together (PRD answer #5, arch §Layout):

  • Widen the panel ~an inch: .panel goes from width: min(60rem, 65vw) to min(64rem, 68vw). The dimmed board stays visible behind it.
  • Constrain the prose to a measure: the single column is capped at max-width: 72ch; margin-inline: auto inside the surface. So even though the panel is wider, the text column is narrower than today's preview pane — landing in the 65–80 char target.
  • The split is gone. No more 1fr / 1fr editor+preview giving each side ~30rem; one centered column gets the room.

The left frame reproduces today's monospace source (line numbers, syntax tokens in --secondary/--tertiary) beside a cramped preview to make the contrast concrete for the builder. Chat still attaches as a column when toggled (Screen 6), which re-narrows the prose — acceptable, since chat sessions are an editing aid, not the reading default.

Library decision

Three candidates were weighed against the existing CodeMirror 6 stack already pinned in package.json:

▦
Option Round-trip fidelity Tables Verdict
CodeMirror 6 + live decorations Excellent — the document is the markdown string No table widgets; renders as styled source Weak on “looks rendered”
Milkdown (ProseMirror + remark) Very good — remark is the round-trip standard GFM table plugin with editing Best match to all three
TipTap (ProseMirror) Good for HTML; markdown is a secondary export Table extension (editing-oriented) Round-trip diffs leak
＋ add row · ＋ add column

Recommendation: adopt Milkdown. A remark-based pipeline is the only candidate whose document model is markdown while still rendering as true WYSIWYG and offering a real GFM table editing experience.

Screen 5 · Tables — render & edit

Tables that read like the VS Code preview

This is the architecture's own library-decision table, rendered the way Plans and rollups need (PRD goal #5). New .markdown table/th/td rules (absent today) bring it up to VS Code-preview quality:

  • Borders: border-collapse: collapse with 1px --outline-variant on every cell — a clean grid, not the hairline-free mush of today's marked output.
  • Header row: --surface-container-high fill, weight 600, left-aligned, white-space: nowrap so column labels don't wrap.
  • Padding & zebra: 0.42rem 0.7rem cells for breathing room; even rows get a faint --surface-container tint for scan-ability. tabular-nums keeps numeric columns aligned.
  • Editing affordance: the focused table node shows a --primary outline, a corner grip handle, and an “add row / add column” control. Because Milkdown's GFM table node reuses the same .markdown-scoped rules, reading and editing look identical.

Builder note: keep the table styling scoped to .markdown so it covers both the editing surface and any non-panel render path. The grip/add-row chrome is editor-only and disappears in read-only/approved docs.

For the planner

Open questions & design flags

  • ✅ RESOLVED — Token source confirmed. These mockups were grounded in the real, shipped Obsidian Flux token layer (plugins/specmanager/ui/src/tokens.css + styles.css) per the architecture. Decision: confirmed correct — this is the right token source. (Note that the repo ./docs/DESIGN.md still holds auto-generated stubs (--primary #1A1C1E); if DESIGN.md is ever meant to be canonical it would need a re-sync to match the live tokens, but the live token layer is authoritative for this work.)
  • New tokens introduced (flagged): nothing new on the color scale — but four style decisions aren't yet in the token system: the toolbar active-state fill (--primary 14% mix), table header fill (reusing --surface-container-high), even-row zebra (--surface-container 55% mix), and the table cell border (--outline-variant). All are derived from existing tokens, but the planner should decide whether to name them (e.g. --tb-active-bg, --table-header-bg) or inline the color-mix.
  • ✅ RESOLVED — Heading control UX. The heading control is H ▾ and opens a menu on click to choose the heading level (H1 / H2 / H3) — it does not cycle on click. Decision: menu, not cycle. Builder: MarkdownToolbar's heading button opens a small level-picker popover and dispatches the chosen level against the current selection.
  • ✅ RESOLVED — Toolbar overflow at narrow widths. At narrow breakpoints, collapse list / link / table / code-block into an overflow ⋯ menu. Bold / Italic / Heading stay as primary, always-visible buttons; the remaining four live behind the overflow menu. Decision: primary trio visible + secondary four in a “⋯” menu. Builder: the ⋯ menu reuses the same ToolbarAction commands as the inline buttons.
  • Read-only hint row. Screen 1 adds a slim “Approved — read-only” hint where the toolbar would be. This is a small net-new element; confirm it's wanted, or just hide the toolbar with no replacement.
  • ✅ RESOLVED — Table on mobile / horizontal scroll. Wide tables (the 4-col decision table) will overflow the <640px full-width panel. Decision: horizontal scroll within the table wrapper is acceptable — no stacking treatment required. Builder: keep the .tbl-wrap as an overflow-x: auto container so wide tables scroll rather than reflow.
  • Normalization diff on first save. Not a visual question, but it affects perceived quality: Milkdown's remark round-trip may tidy bullet glyphs / table pipes on the first save of a legacy doc. The arch's Plan-phase round-trip check covers this; the designer assumes the visible result is unchanged.
