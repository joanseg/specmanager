Publish & share

NOT PUBLISHED

Publish a read-only snapshot of this board to your own Netlify account, behind a secret link. No login for readers — open in any browser.

  1. Create a Netlify personal access tokennetlify.com → User settings → Applications → New access token
  2. Run /specmanager-publish ⧉First run asks for the token, creates the site, and deploys. The token is stored outside the repo and never committed.
  3. Copy the secret link and send it to your stakeholdersRe-run the command after specs change — the URL stays the same.
Heads up: published pages live on the public internet behind an unguessable URL. Anyone who has the link can read them.

Screen 2Share popover — never published (first-run / credential setup state)

Same trigger, same popover shell, different body: when GET /api/publish/status returns null the panel becomes a three-step onboarding card instead of a settings form. The board deliberately does not collect the Netlify token — credentials are collected in-session by /specmanager-publish (Architecture flow §5, step 1) and stored in ${'{'}SPECMANAGER_PLUGIN_DATA{'}'}/publish-credentials.json, so the local-only, no-auth board never handles a secret.

  • Status chip flips to a hollow grey "NOT PUBLISHED" — the same dot atom, drained of the emerald glow.
  • Step 2's command chip is the single call to action; copying it and pasting into the Claude Code session is the entire interaction, mirroring the board's existing generate affordances.
  • The PRD friction-proxy metric ("never published → stakeholder reading in one sitting") is what this card optimizes: token source spelled out, URL-stability promise stated up front.
  • The caveat footer appears even before first publish so the "public internet behind an unguessable path" assumption (PRD) is consented to, not discovered.

Screen 3Doc panel — per-doc share strip

The PRD's second sharing mode: "alternatively user can share only individual documents." When the open doc has an entry in publish.json → docs, a one-line share strip renders between the panel header and the doc surface — same slot and geometry as the existing stale banner (panel__stale), but tinted with 7% --primary instead of error rose.

  • Contents, left to right: "Shared" label with live dot · the /d/<docToken>/ URL (its own token — handing out one doc never reveals the board URL, per the Architecture's URL scheme) · a version-drift note · Copy.
  • Version drift is amber (--status-draft) when publish.json records an older version than the doc's current version — "shared v1 · now v2 — re-publish to update". When in sync the note reads "shared v2 · current" in emerald. This is the PRD Q5 answer surfaced to the publisher, not just the reader.
  • Docs with no share entry show no strip at all — sharing a single doc is done via /specmanager-publish <docId> from the session (no mutating board route in v1); the board share popover (Screen 1) lists all existing per-doc shares.
  • Interview-kind docs never show the strip: the Architecture excludes them from publishing unconditionally.

What opens at /s/<boardToken>/. Static HTML, zero client-side JS (Architecture §5d) — every doc card is a plain <a> to /s/<boardToken>/docs/<docId>/. It keeps the product's Obsidian Flux skin (templates lift values from ui/src/tokens.css) so the snapshot is recognizably the same product, but the layout is deliberately simpler than the working board:

  • Per-feature sections instead of the kanban grid. Stakeholders read top-to-bottom; the working board's locked/generate/optional cells, slash-command affordances, and BUILD column are publisher concepts and are gone. Missing stages render as quiet dashed placeholders so the lifecycle shape stays legible without explaining gates.

  • <meta name="robots" content="noindex"> on every page; the secret lives only in the URL path, shown in the browser chrome above.
  • The "made with SpecManager" wordmark is restrained mono in the footer — attribution without branding the stakeholder's content.

Based on PRD prd-dunning-emails-007 v2. We add a retry scheduler to the billing worker and a templated email pipeline rendered from the invoice state machine.

1. Summary

Failed charges enter a three-step retry ladder. Each step emits a templated email through the existing notification service; the final step downgrades the subscription and notifies the account owner.

2. Retry ladder

StepWhenEmailOn failure
1T+3 daysPayment reminderschedule step 2
2T+7 daysPayment failed — action neededschedule step 3
3T+14 daysFinal noticedowngrade + owner alert

3. Affected components

  • billing/worker.ts — retry scheduler, idempotent per invoice
  • notifications/templates/ — three new MJML templates
  • core/subscription.ts — downgrade transition guarded by the ladder state

Screen 5Published document page — draft doc inside the board snapshot

One markdown doc rendered through the remark/rehype pipeline into the same template shell. The reading column matches the board's .prose rules (0.95rem / 1.62 body, Hanken Grotesk headings) but capped at ~46rem — stakeholders get a document measure, not the editor's 90% panel width.

  • Draft flag sits beside the title — bordered amber chip, louder than the board's tiny badge because here it is the one piece of workflow context a stakeholder must not miss. Approved docs get the equivalent emerald "Approved" chip.
  • Version line appears twice: under the title and as the footer the Architecture specifies verbatim — v<version> · <updatedAt> · snapshot published <timestamp> (PRD Q5: show the version being seen).
  • Navigation: "← All features" + a mono crumb. This top bar only renders for board-snapshot doc pages. An individually shared doc at /d/<docToken>/ uses this exact template minus the back link and crumb — linking back would leak the board token, which per-doc shares must never do.
  • Tables get the board's markdown-viewer styling (header on --surface-container-high, zebra rows); raw HTML in markdown is dropped by the pipeline (deliberate sanitization). Design-brief docs ship their self-contained HTML verbatim as the page body instead of this prose shell.
  • Zero JS; every style inlined in the page head by publish/templates.ts.

Screen 6404 — unpublished snapshot, site root, or wrong path

The single 404 page serves three cases identically: a snapshot that was unpublished (the deploy replaces all content with just this page; siteId/baseUrl survive so a later publish returns to the same URL), the bare site root /, and any guessed path.

  • Deliberately information-free: no project name, no doc list, no SpecManager wordmark, no hint that valid tokens exist. "No longer shared" is the only concession to the legitimate visitor holding a stale link — it tells them to ask the publisher for a fresh one without confirming anything to a guesser.
  • Same dark canvas and type so a stakeholder who had the site before doesn't think they've landed somewhere malicious.
  • Mono "404", display-face message, dim sub-line — three lines, centered, nothing else.

For the plannerOpen questions

  • Unpublish command syntax. The Architecture defines the unpublish_snapshot MCP tool but only one slash command (specmanager-publish.md). Screen 1 mocks /specmanager-publish unpublish as an argument form — confirm whether unpublish is an argument, a separate command, or conversational only, and update the popover chip text accordingly.
  • Fonts in the static snapshot. The board self-hosts Hanken Grotesk / Inter / Geist Mono via fontsource. Screens 4–6 assume the snapshot ships the same woff2 files inside the zip (a few hundred KB, keeps zero-external-asset posture). If the planner prefers a leaner zip, the system-ui fallbacks already in the token stacks degrade acceptably — decide one way.
  • Dark-only stakeholder pages. The published site inherits the product's dark theme (templates lift tokens.css). No light variant exists in the design system; if stakeholder feedback demands one it is a new token set, out of v1.
  • Tokens not in DESIGN.md. The DESIGN.md frontmatter still holds placeholder var() references; the canonical values used here come from ui/src/tokens.css (the file the Architecture itself points templates at). New, named in this design: the share-strip tint (7% --primary color-mix, same recipe family as existing banners) and the bordered draft/approved flag chips on published doc pages (derived from --status-draft/--status-approved at 14%/12% background mix). No new raw hex anywhere.
  • Share popover live updates. Screen 1 assumes the popover refreshes on snapshot.published/snapshot.unpublished WS events (the Architecture already broadcasts them). If the popover is open during a publish from the session, the URL/status should update in place — cheap, but worth a task line.
  • Relative timestamps. "2 h ago" in the header pill implies a relative-time formatter on the board; the popover and all published pages use absolute timestamps. If the planner wants zero new utilities, the pill can show the absolute date too.
