Antigravity plugin — experience mockups

This feature ships almost no new board UI — the board is explicitly unchanged in the architecture. What it ships is an experience on a third-party surface: Antigravity's agent panel, driven by our generated workflows, rules file, and INSTALL.md bootstrap prompt. The "UI" we control there is the text the agent emits — so screens 1–3 are format contracts for prompt output, rendered as stylised, clearly-labelled illustrations of Antigravity (not pixel-accurate Google UI). Screen 4 is the single real board change (the unsupported-chat disclosure — the in-UI chat is not supported on Antigravity, by user decision 2026-06-11). Screen 5 shows the AGENTS.md managed block contract.

All SpecManager board surfaces below use the real Obsidian Flux tokens from ui/src/tokens.css. One shared color language carries across both surfaces: amber = waiting/needs you, emerald = done/approved, rose = failed/stale — the same status roles the PM will learn on the board.

Screen 1 · Terminal-free install — bootstrap prompt → verified "board is up" moment

Screen 1 — The install experience (happy path)

This is the PM success metric made visible: install without Joan, without a terminal. The user performs exactly two actions — paste the INSTALL.md bootstrap prompt, and one toggle in the MCP store — and the agent does everything else.

  • This screen is a format contract, not a UI we render. The deliverable is the wording of antigravity/INSTALL.md: the prompt explicitly instructs "report each step as pass/fail with one line of evidence", which produces the numbered checklist above. The builder implements this in the INSTALL.md copy and reinforces it in the generated rules file.
  • Every step = name + state icon + one line of mono evidence. Evidence is specific and checkable (commit hash, package count, node version, tool count, HTTP 200) — a semi-technical PM can read it; Joan can debug from it remotely.
  • The one manual step is designed, not apologised for. The MCP-store toggle is the only thing the agent can't click (it's Antigravity's own chrome), so it gets a distinct amber "action needed" card with the exact location of the control, and the flow resumes only after the user confirms. Step 4 then re-verifies (tool count) rather than trusting the user.
  • The "board is up" moment is the climax — an emerald card with the URL as a button and the immediate next action (/specmanager-prd). Install ends with momentum, not a log dump.
  • State colors deliberately mirror the board's status tokens (--status-draft/--status-approved/--status-stale roles): amber, emerald, rose. One color language across both surfaces.
  • Architecture grounding: installer behaviour shown (literal paths in mcp_config.json, preserve other servers, copy .agent/workflows|prompts|rules) matches antigravity-install.ts exactly — the mockup's evidence lines are the installer's required stdout.

Screen 2 · Install failure — root cause + proposed recovery, never a raw log dump

Screen 2 — Install failure & recovery

The PM refuses terminals; a wall of npm stderr is the moment we lose him. The failure contract INSTALL.md must encode:

  • The checklist stays visible — failed step goes rose, downstream steps stay pending (hollow), so the user always knows where it stopped and that nothing after it ran ("nothing was half-installed" is stated explicitly — install must be safe to re-run, which the architecture's installer guarantees via merge-not-overwrite).
  • Root cause first, in one sentence, then exactly one proposed fix with an approval choice. The raw log is available on request, never shown by default.
  • Recovery resumes, not restarts — "resume from step 2" reinforces that steps are idempotent and ordered.
  • The two buttons are Antigravity's native command-approval affordance (illustrative rendering); the real mechanism is the agent proposing a command and the IDE asking to run it. // turbo annotations in workflows can auto-approve safe commands later, but the install flow keeps explicit approval — first-run trust matters more than speed.

Screen 3 — Drafting from Antigravity (dual surface)

The core loop: type /specmanager-prd in the IDE, watch the card appear on the board. The two windows are the product — Antigravity is where drafting happens, the board is where the human reviews, approves, and gates open. Nothing on the board side is new; it must look exactly like today, which is itself the design statement (artifact parity, PRD metric 2).

  • Workflow provenance chip: the generated workflow's path (.agent/workflows/specmanager-prd.md) is surfaced so issues are debuggable ("which prompt ran?"). This comes free from Antigravity; our generated workflow's opening line should restate it.
  • MCP tool rows show projectDir explicitly — the rules-file contract ("always pass the workspace root") made visible. The generated workflows should instruct the agent to narrate tool calls in this compact tool(args) → result form rather than prose.
  • The closing card teaches the gate model: "I can't approve docs; Architecture stays locked until you do." Enforcement is in core (a Gemini agent ignoring its prompt still can't pass check_gate), but the workflow copy keeps the human's mental model right.
  • Board side, real tokens throughout: card gradient (--card-gradient), draft badge --status-draft amber, the new card gets the hover-tier violet border (--primary mix) as a "just arrived" highlight tied to the doc.created WS pulse in the header — this matches existing card/hover styling, no new component.
  • Locked / optional / empty cells render exactly as the existing LockedCell / OptionalDesignCell / EmptyCell components — shown here so the planner sees the gate states an Antigravity-driven repo flows through.

Screen 4 · Unsupported-feature disclosure — chat is not part of the Antigravity port, said honestly

Screen 4 — The unsupported-chat disclosure (the one real board change)

By user decision (2026-06-11, written acceptance in PRD success metric 3), the board's in-UI chat is not supported on Antigravity — the scorecard entry is unsupported, and no port work targets it. Today ChatPanel.tsx renders a terse "Chat unavailable. no API credential found"; for the trigger PM that reads like breakage. This screen upgrades that one state to say plainly that chat is outside the port's scope — with no "add a key" call-to-action (we don't upsell a feature we've declared unsupported).

  • Same trigger, same component: the existing /api/chat/status → available:false path and the .chat--unavailable branch — only its rendered content changes. No new API, no detection of "is this an Antigravity user" (no such signal exists; the copy works for everyone).
  • Tone: off, not broken. Amber identity icon (not rose — nothing failed), a plain reason, then the reassurance in bold: everything else works without it, with the concrete alternative (draft from the IDE, edit in Milkdown here).
  • Honesty footer in mono shows the actual status-check result — consistent with the evidence-line pattern of screens 1–3, and Joan-debuggable.
  • Components/tokens: standard card (--card-gradient, --card-border, --radius-md), single secondary .btn (no primary CTA — unsupported features get no action button), composer rendered disabled at 50% opacity. "Dismiss" collapses the card to a one-line note for the session (no persistence needed).
  • The drawer, dimmed board behind (--backdrop-dim), and editor column are today's UI verbatim — drawn here only for context.

Screen 5 · AGENTS.md — the same managed block as CLAUDE.md, between the same markers

Screen 5 — The AGENTS.md managed block in context

Antigravity prepends AGENTS.md to every prompt the way Claude Code reads CLAUDE.md, so the lifecycle block must live there too. This screen is a content contract, not UI — the violet highlight and "managed by" chip are mockup annotations (no editor renders them); what's real is the text.

  • Identical markers, identical block: <!-- specmanager:start --> / <!-- specmanager:end -->, same content as the CLAUDE.md block, produced by the same line-anchored merge (core/claude-md.ts → syncAgentsMd). User content above and below the markers is never touched — shown explicitly with user lines on both sides.
  • One adaptation: the commands line reads "Workflows" and lists the /specmanager-* workflow names — the same names, so a dual-client repo's block is valid for both. (If keeping the literal word "Commands" simplifies sharing one block, that's acceptable — flagged below.)
  • Per the architecture, sync writes AGENTS.md only if the file already exists — pure Claude Code projects never grow one.

Open questions for the planner

  • Token source of truth: docs/DESIGN.md frontmatter is thin (one body-md role, indirect color values), so these mockups are grounded in ui/src/tokens.css (Obsidian Flux) as the de facto system. No new tokens were invented; the unsupported-chat card uses only existing roles. DESIGN.md should eventually be re-synced from tokens.css, but that's outside this feature.
  • ✅ RESOLVED — chat is unsupported on Antigravity (user decision, 2026-06-11): the former "How to add a key" CTA is removed from Screen 4; the card states chat is not part of the port and offers only Dismiss. No destination link is needed.
  • Dismiss persistence: Screen 4's Dismiss is session-only (collapses to a one-liner). If the PM finds it nagging in the supervised session, consider localStorage persistence — defer until observed.
  • MCP-store toggle wording (Screen 1, step 4): the exact location/label of Antigravity's MCP enable control must be verified in the Phase A spike and the INSTALL.md copy updated to match the real build — the mockup's "plug icon, top right" is a placeholder.
  • Agent output fidelity under Gemini: screens 1–3 are format contracts carried by prompts; Gemini may drift from the checklist/tool-row format. The Phase A exit test should check the shape of the output (per-step evidence lines present), not exact rendering, and fixes go into commands/*.md / INSTALL.md — never hand-edited generated files.
  • AGENTS.md block sharing: is one byte-identical block for CLAUDE.md and AGENTS.md acceptable (simplest: keep the "Commands:" label verbatim), or does AGENTS.md get the "Workflows:" variant shown in Screen 5? One block is less code; the variant is clearer for Antigravity users. Recommend the variant only if the merge helper already parameterises content trivially.
  • Event-pulse label: Screen 3 shows the raw event type (doc.created) in the board header — that's today's behaviour, kept deliberately. No change requested.
