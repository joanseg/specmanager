SpecManager for Cursor

cursor ≥ 0.50 node ≥ 20 local-only · 127.0.0.1

SpecManager turns your feature lifecycle (PRD → Architecture → Design → Plan → Build → Walkthroughs) into a localhost kanban board backed by plain markdown in your repo. This guide installs the Cursor delivery surface; the board, gates, and artifacts are byte-identical to the Claude Code plugin — a repo can be driven from either client interchangeably.

1Get SpecManager

# keep the checkout somewhere stable — .cursor/mcp.json will point at it by absolute path
git clone https://github.com/joan/specmanager ~/tools/specmanager
cd ~/tools/specmanager/plugins/specmanager/server && npm install

2Install into your project

Option A — installer script (recommended) 
node ~/tools/specmanager/plugins/specmanager/server/dist/cursor-install.js \
  --project /absolute/path/to/your/project

The installer merges a specmanager entry into your project's .cursor/mcp.json (creating it if missing, preserving any other servers), copies the 9 commands into .cursor/commands/, the rules file into .cursor/rules/, and the trampoline prompts into .cursor/specmanager/prompts/. It is idempotent — re-run it any time.

Option B — let Cursor's agent do it

Paste the bootstrap prompt from the bottom of this file into Cursor's agent panel; it runs the same installer with your workspace path filled in.

! One machine-specific file. .cursor/mcp.json carries absolute paths (the SpecManager checkout and your project root). Commit everything else in .cursor/ so teammates inherit the commands; after cloning, each machine re-runs the installer once.

3Verify

  1. 1Reload Cursor (Cmd+Shift+P → "Reload Window") and enable specmanager under Settings → MCP.~20 tools listed
  2. 2Type /specmanager- in the agent input.9 commands autocomplete
  3. 3Ask the agent to "initialise SpecManager" (or run /specmanager-init)..claude/specs/ scaffolded
  4. 4Open http://127.0.0.1:4317.board loads

Troubleshooting

Tools fail with "could not find .claude/specs"
The project root didn't resolve — re-run the installer so SPECMANAGER_PROJECT_DIR matches your workspace, then reload Cursor.
Cloned the repo on a new machine and commands stopped working
.cursor/mcp.json is the one machine-specific file. Re-run the installer; it merges, never overwrites.
Board port already in use
Another client (e.g. Claude Code) already runs the board. That's fine — the second instance defers via the pidfile takeover logic. Both clients share one board.
Board chat says "Chat unavailable"
Expected. The in-board chat needs Anthropic credentials and is supported in Claude Code sessions only. Drafting and edits work from Cursor via the /specmanager-* commands.

Screen 1 — INSTALL.md: the self-serve story

This is PRD success metric 1 made tangible: a Cursor user must complete the install without Joan's help. The doc is structured as exactly three numbered steps plus troubleshooting — no branching narrative, one recommended path (Option A), the agent-bootstrap path (Option B) as a one-liner.

  • Decisions a builder must honour: the amber callout about .cursor/mcp.json being the one machine-specific file is load-bearing (architecture "Failure & edge cases" names this exact failure); the verification list mirrors, word for word, the checklist the installer prints (Screen 2) so the doc and the terminal never disagree; the chat-unavailable troubleshooting entry pre-empts the one "is this broken?" moment a Cursor user will hit (Screen 5).
  • Each verify step has a machine-checkable expectation (right-aligned, emerald --success mono) — pass/fail, not vibes, matching the coverage-scorecard discipline.
  • Tokens: frame and code blocks on --surface-container-low/lowest, hairline --outline-variant, step numbers and inline paths in --primary, warning callout on --tertiary, radii --radius-md/lg. Headings in the display face, body at 14px.
  • This mockup defines the doc's structure and tone; the markdown source ships in plugins/specmanager/cursor/INSTALL.md and renders wherever the user reads it (GitHub, Cursor's preview) — the visual treatment here shows intent, not a CSS deliverable.

Screen 2 — installer output: merge report + printed checklist

The architecture specifies that cursor-install.ts "prints the verification checklist" — this screen is that contract. The output reads in three beats: what I touched (four check lines), the one thing you must know (amber machine-specific warning), how you prove it worked (the same four-step checklist as INSTALL.md, with explicit expect: lines).

  • The merge line says what it preserved ("1 existing server preserved: linear") — this surfaces the selftest's clobber-protection guarantee to the user, building trust in the idempotent re-run story.
  • ANSI mapping: checks in green (≈ --success #10b981), the machine-specific warning in yellow (≈ --tertiary), the brand/section headers in the terminal's blue/magenta (≈ --primary/--secondary), paths and asides dim. Use standard 16-color ANSI (not truecolor) so it survives any terminal theme; the hues here are the design intent, not literal escapes.
  • Failure variants (same layout, builder implements): missing --project or a relative path → one rose ✖ line + usage; unwritable .cursor/ → ✖ on that row, remaining rows skipped, non-zero exit. Errors keep the column alignment of the success rows.
  • No spinner, no progress bar — the whole run is sub-second; print the report once, exit.

Screen 3 — command discovery and the draft loop (with concurrency recovery)

Left: typing /specmanager- surfaces all 9 generated commands with one-line descriptions — this is verification step 2 rendered, and the descriptions are a generator output (first line of each command file), so they must be written for this dropdown: verb-first, under ~6 words.

Right: the full draft loop under a non-Claude model (the model pill deliberately shows GPT — Cursor's population runs mixed models, and the prompts must hold either way). Three designed behaviours:

  • Every SpecManager tool call shows projectDir on its first row — the rules file (specmanager.mdc, alwaysApply) mandates passing the workspace root per call; making it visible in the transcript is how a user (and the spike) confirms the rules are being obeyed.
  • The closing message has a fixed shape: doc id + status + version, the "I don't approve documents" line (rules contract), board link, and the named next command. Builders should bake this shape into the generated command epilogues so every stage ends the same way.
  • The baseVersion rejection is shown, not hidden: rose ✗ row with the literal reason (baseVersion 1 ≠ current 2), then re-read → re-apply — the exact recovery move the rules file prescribes. The agent's explanation names whose edits won ("your goal edits intact"). This is PRD metric 2's concurrency guarantee made visible.

Chrome caveat: the panel framing (header, bubbles, input) is illustrative of Cursor's own UI, which we do not ship. The designed artifacts here are the command names + descriptions, tool-call sequencing, argument surfacing, and the agent copy — all of which come from our generated prompt files.

Screen 4 — the highest-fidelity-risk flow, made legible

Build is the flow the PRD flags as riskiest under a flattened, non-Claude agent (A2/A3). The design counters with structure the user can audit at a glance:

  • The trampoline is visible and labelled. The first action after /specmanager-build is a file read of .cursor/specmanager/prompts/builder.md, tagged trampoline (violet, --secondary family — a mechanism label, not a status). The user learns the read is expected, not the agent wandering.
  • One phase card, tasks in dependsOn order, each with its Fibonacci score and an emerald tick as update_task lands. The card is the agent's rendering of get_next_phase output — generated command prompts should instruct the agent to print exactly this checklist shape before working.
  • The stop is attributed to the server, not obedience: the final get_next_phase row shows the boundary ("phase C exists — not started"), and the closing message says "the server won't hand me Phase C in this run." This phrasing matters — it teaches that gates are enforced in core, which is exactly why a weaker model can't drift past them.
  • The closing message always names both next moves (walkthrough vs next build) with the next phase's name and task count, so the lifecycle stays self-navigating without the Claude Code docs.

prd draft v3

Problem

Stripe webhook deliveries fail silently when the checkout service is mid-deploy; events are dropped with no replay path, and reconciliation is manual…

Goals

1. No webhook event lost across deploys.  2. Automatic retry with exponential backoff.  3. Dead-letter visibility within 5 minutes…

Success metrics

Zero dropped events over a 7-day window with two forced deploys; replay completes within 60s of recovery…

Screen 5 — the one board state a Cursor session changes (and how little it changes)

The architecture pre-accepts the in-board chat as unsupported on non-Claude clients and says "the chat panel's existing available: false state renders with plain copy; no port work targets it." This screen shows that existing state (the real .chat--unavailable styles: rose --error heading, dim body, no new components) in the context a Cursor user will meet it — everything else on the doc panel works normally: badges, history, the gradient Approve button, live doc content.

  • One copy change is proposed (and only one): today the reason renders as the bare server string ("no API credential found"). The mockup appends a clarifying clause plus a dim hint line redirecting to the editor agent. This is a string-level change in the chat-status reason / panel copy, not a component change — but since the architecture scoped zero board work, it needs an explicit scope call (see Open questions).
  • Nothing else on this screen is new. Header, badges (badge--draft amber, badge--meta), split layout, and the Approve CTA (--gradient-primary) are the shipped board, included so the planner sees the unsupported state is a corner, not a wall — the coverage scorecard marks exactly one cell "unsupported", and this is it.

Open questions for the planner

  • Chat-unavailable copy (Screen 5): the friendlier reason + hint line is a one-string-ish change touching the board (server reason text or panel copy), which the architecture scoped to zero. Accept as a deliberate one-line exception, or ship the bare existing string and keep the friendly copy in INSTALL.md's troubleshooting only? Either is fine; decide explicitly.
  • Cursor chrome is illustrative: Screens 3–4 depict Cursor's own panel, which we don't ship. The binding deliverables are the command names + one-line descriptions (dropdown), the projectDir-on-every-call rule, the closing-message shapes, the phase-checklist rendering instruction, and the baseVersion recovery sequence — all of which live in the generated .cursor/ prompt files. Spike item 2 should confirm a non-Claude model actually reproduces these transcript shapes.
  • Command descriptions as generator output: the autocomplete one-liners (Screen 3a) assume Cursor surfaces a description per command file. If it only shows filenames, the dropdown degrades gracefully (names alone are self-describing) — verify in the spike and note it on the scorecard.
  • Terminal palette has no DESIGN.md token: Screen 2 maps checks/warnings to standard ANSI colors that approximate --success/--tertiary/--error. DESIGN.md has no terminal-output guidance; if more CLI surfaces appear, consider adding a "CLI output" section to docs/DESIGN.md.
  • DESIGN.md frontmatter lags tokens.css: the managed frontmatter still lists placeholder values (system-ui body-md, var() color indirections), while ui/src/tokens.css carries the real Obsidian Flux scale (Inter/Hanken Grotesk/Geist Mono, hex palette). These mockups follow tokens.css as ground truth; the next sync_design_md pass should close that gap.
  • Installer version string: Screen 2 shows "installer v0.9.0" — there's no established version source for the installer banner. Use the server package.json version, or drop the version from the banner.
