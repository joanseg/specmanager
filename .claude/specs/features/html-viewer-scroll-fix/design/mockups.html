Screen 1 — what's broken today

This is the existing state the fix targets. The design brief renders verbatim HTML inside a sandboxed <iframe class="panel__preview panel__preview--iframe" sandbox="allow-same-origin"> in DocPanel.tsx. The iframe is styled width/height: 100% but has no display: block (styles.css ~L817), so as a replaced inline element it sits on the text baseline and leaves a few-px descender gap below it. Against a height: 100% iframe in an overflow: hidden grid cell, that gap forces the wrapper to overflow — producing the spurious horizontal scrollbar at the bottom plus an out-of-parity vertical one.

  • Two scrollbars, native OS styling — no ::-webkit-scrollbar or scrollbar-color rules exist anywhere in ui/src/ today, so this viewer doesn't match the markdown one even on the vertical axis.
  • Tokens shown: drawer chrome on --surface / --outline-variant; iframe document renders its own dark Obsidian Flux body (real SpecManager briefs are dark — --design-preview-bg #ffffff is only the fallback behind the iframe, visible here as the thin white sliver in the descender gap); DESIGN badge on --secondary-container / --on-secondary-container; APPROVED on --success.
  • This screen is the "before" reference for the builder — it is the regression to eliminate, not a target.

Screen 2 — the fixed viewer

Same drawer, after the fix. Three changes land together:

  • Kill the descender gap. Add display: block to .panel__preview--iframe. The iframe now fills its grid cell with no baseline space below it, so the wrapper never overflows and the spurious horizontal scrollbar disappears.
  • Reset inside the iframe. Prepend a <style>html,body{margin:0}</style> block to srcDoc so the verbatim mockup body has no default 8px margin fighting the canvas. Parent CSS cannot reach inside a sandboxed iframe, so this must be injected.
  • Unified scrollbar (width + colour). The same injected <style> carries the custom-scrollbar rules so the iframe document's vertical scrollbar matches the markdown surface — scrollbar-width: thin + scrollbar-color for standards engines, ::-webkit-scrollbar for WebKit.

Scrollbar tokens (proposed): thumb --scrollbar-thumb = --outline-variant (#464554), hover --scrollbar-thumb-hover = --outline (#908fa0), width 10px, transparent track. All derived from existing tokens — no new colour. The sandbox="allow-same-origin" attribute and the --design-preview-bg fallback are preserved exactly — but because the brief paints its own dark body and html,body{margin:0} removes the gap, that white fallback is no longer visible.

PRD APPROVED

Problem

The board's document panel renders two kinds of content in the same drawer, but they do not scroll the same way.

Goals

The HTML/design viewer should scroll like the markdown viewer: a single, consistent vertical scroll, with scrollbar-color matching across both surfaces.

Constraints

Keep the iframe sandboxed. Use the existing Obsidian Flux design tokens. No new ad-hoc colours.

The prose column fills 90% of the panel width and never overflows horizontally — the reference behaviour this fix matches.

Screen 3 — the markdown reference, brought into the same scrollbar family

The markdown viewer (.md-surface, flex: 1; overflow: auto) is the reference: its prose column is capped at width: 90% so it only ever scrolls vertically. Its behaviour is a non-goal to change — but PRD open-Q2 ("width and colour across both viewers") means the same custom-scrollbar styling is added here too, so the two drawers feel identical.

  • The shared rule is applied via a scoped utility (e.g. .scroll-surface) on the two named surfaces — never a global * rule — to avoid regressing anything else in the board.
  • Both surfaces are dark, so the thumb's faux-padding border tracks the surface it sits on — --surface on the markdown column, --surface-container-lowest on the iframe document. Same thumb colour and width across both.
  • No change to MarkdownEditor behaviour or the 90% column — only the scrollbar appearance is added.

Screen 4 — the wide QA fixture (PRD open-Q3 deliverable)

PRD open-Q3 was answered "construct one on design phase," and the Architecture flags the lack of a wide brief to test against. This screen is that fixture. Its inner content is a fixed ~1680–1760px layout — genuinely wider than the ~64rem drawer.

  • Expected behaviour: horizontal scroll appears inside the iframe document only (intentional, per PRD open-Q1 answered "correct") — the .panel__preview--iframe wrapper itself must not show a wrapper-level horizontal scrollbar, and the drawer must not clip or rescale (no zoom/fit-to-width — PRD non-goal).
  • How to use it: the builder opens this very mockups.html in the board's design viewer. Screen 4b is a real horizontally-overflowing strip — if the fix is correct, that strip scrolls sideways within the iframe and the surrounding drawer stays put. If the wrapper itself grows a horizontal bar, the descender/overflow regression is back.
  • The distinction the fix must preserve: spurious wrapper-level horizontal scroll (the bug — eliminate) vs. legitimate content-wider-than-viewport scroll inside the iframe (intentional — keep). The custom scrollbar styling applies to this horizontal bar too (matching height + colour), so even the intentional scroll looks on-brand.

Open questions for the planner

  • DESIGN.md stub vs. real tokens. The repo's ./docs/DESIGN.md frontmatter is still the auto-generated placeholder (primary: #1A1C1E, one body type role). These mockups are grounded in plugins/specmanager/ui/src/tokens.css ("Obsidian Flux"), whose header states it maps 1:1 to the real DESIGN.md. Confirm tokens.css is the canonical source for this feature, and that DESIGN.md will be re-synced — no design decision here depends on the stub.
  • New scrollbar tokens — name or inline? No scrollbar token exists today (Architecture open-Q1). This brief proposes --scrollbar-thumb (= --outline-variant), --scrollbar-thumb-hover (= --outline), and --scrollbar-size: 10px. These reuse existing colours (no new colour) but --scrollbar-size: 10px is a new dimension value not currently in tokens.css. Planner to decide: add semantic aliases to tokens.css, or inline the existing tokens directly. Flagging the 10px as the one non-token value introduced.
  • Scrollbar width target. 10px is a proposal for a comfortable, on-brand thin bar. Confirm the exact width (8px aligns to the 8px spacing grid; 10px reads slightly more present). Whatever is chosen must be identical on both surfaces and on both axes.
  • Injection mechanism. Architecture recommends prepending the <style> to srcDoc (vs. an onLoad writer). Mockups assume the srcDoc-prepend. Confirm.
  • Where the QA fixture lives. This mockup file doubles as the fixture (Screen 4b). Should a standalone throwaway wide brief also be created under a _qa feature, or is opening this mockups.html in the viewer sufficient for manual verification?
