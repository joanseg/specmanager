Screen 1 — PRD column with the Interview chip (recommended treatment)

The full board, three rows covering the three states that matter:

  • Row 1 (PRD + interview): the normal DocCellView PRD card keeps its pinned 5.5rem height; the chip renders beneath it inside a new .cell-stack column wrapper. Because .row__cell is already align-items: flex-start, the slightly taller cell pins to the top of the row track and nothing else moves.
  • Row 2 (interview, no PRD yet): the interview-first flow. The chip sits beneath the existing card--ready generate affordance — exactly the Architecture's "Interview exists, PRD doesn't" edge case. Row 2's chip is shown in its frozen hover state (border tints toward --card-border-hover, text lifts to --on-surface).
  • Row 3 (no interview): chip absent — the column reads identically to today, so the optional artifact costs nothing when unused.

Chip anatomy: pill (border-radius: 999px — the only fully-round element on the board, which is what makes it read as "not a card"), --surface-container-low fill, --card-border hairline, --font-mono at 0.68rem like other meta text, label "Interview" plus a dimmed · vN. The leading hollow ring in --primary deliberately inverts the status pills' glowing filled dot: filled = lifecycle status, hollow = reference material with no status. No new color enters the scale.

Interactions: the whole chip is a button wired to the existing onOpenDoc(id); title carries the doc id like the cards do. No badge, no stale state, no approve affordance — interviews are outside the staleness graph by contract.

Screen 2 — Resolving Architecture open question #1: chip vs stacked card

Both options rendered against the same PRD card at true cell width.

Recommendation: Option A, the chip. Card shape on this board means "stage document with a lifecycle"; the interview has no lifecycle — it is upstream raw material. A pill (the board's only fully-rounded element) plus the hollow ring gives instant visual separation at a fraction of the vertical cost, and keeps the columns' one-card scan rhythm intact. Interview — Interview command

Mode: startup interrogation → switched to builder mode at turn 9 (internal tooling; demand evidence is your own dogfooding history). Method credit: gstack office-hours forcing questions.

Context extracted

  • Problem: the lifecycle starts at /specmanager-prd with a one-shot prompt; the richest context never leaves the orchestrator's head and gets re-injected during review churn.
  • Users: the orchestrator (terminal + board); the prd-writer as indirect consumer of the stored artifact.
  • Constraints surfaced: subagents are single-shot — a multi-turn interview must run in the main session; all persistence through core via MCP tools.

Critique (forcing questions)

  • Demand reality: claimed pain is PRD review churn. Evidence: 3 of the last 5 PRDs in this repo needed 2+ human edit passes. Real, but thin — track it.
  • Status quo: today you paste a long prompt and fix the draft on the board. Cheap. The interview must beat "just type more" or it dies of friction.
  • Narrowest wedge: terminal-only chat, one markdown artifact, zero board UX beyond display. Agreed during the session — board-side interviewing was cut.

Recommended wedge

Ship the conversation quality first (phase 1 go/no-go), persistence and board chip second. If the chat is not better than typing a long prompt, stop.

Screen 4 — Terminal interview flow (reference for the command prompt, not board UI)

This is a stylised reference for the textual UX that commands/specmanager-interview.md should produce in the Claude Code session. Nothing here is rendered by the board; the builder uses it as the target shape of the conversation, not as a screen to implement.

  • Opening turn: goal, the numbered interview plan (6 one-line areas, current area highlighted), and the exit phrase — stated up front, restated roughly every five turns.
  • Plan diff, not plan dump: when the plan changes, print a short + added / − dropped (reason) block — never re-print the whole plan as a wall of text.
  • Critique woven in: the "Hold on — my teammate" beat shows the forcing-question style: challenge the claim the moment it appears, then fold it into the plan.
  • Early exit honored instantly: "finish interview now" (and obvious paraphrases) goes straight to synthesis — no "are you sure", however thin the material.
  • Synthesis block: Extracted / Critique / Recommended wedge / Unresolved — the same four sections the stored markdown body uses (Screen 3), so the chat and the artifact match.
  • Storage prompt names the consequence: "It will appear in the PRD column on the board" — connecting the terminal act to the board surface; on yes, report id + path like every other command.
