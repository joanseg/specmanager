# SpecManager — Architecture & Specification

> A Claude Code plugin that manages a software project's whole lifecycle —
> PRD / press-release → architecture → plan → tasks → build → walkthroughs — through a
> React/Vite kanban board, with Claude Code as the drafting engine and the human
> as editor/approver. *"SpecManager" is a working name; rename freely.*

**Status:** v0 design · **Scope:** single-user, fully local · **Source of truth:** git-tracked markdown

---

## 1. Goals & non-goals

**Goals**

- Make the software-project lifecycle visible and navigable as a kanban board.
- Keep every artifact (PRD, press-release, architecture, plan, tasks, walkthroughs) as plain markdown in the repo, so it diffs, reviews, and travels with the code.
- Track multiple **features**, each flowing through the lifecycle; let Claude Code *draft* each stage from the previous approved stage **and the existing codebase**, and let the human *edit and approve* in the UI.
- Treat approved docs as **living**: revising one flags its dependents for review rather than silently going stale.
- One shared logic layer used identically by the agent (via MCP tools) and the human (via the UI), so the two never drift.
- Live updates: when Claude changes a doc or task, the board reflects it; when the human edits in the board, Claude is made aware.

**Non-goals (v0)**

- Multi-user / real-time collaboration, auth, or a hosted backend (single-user local only).
- Replacing git as version history — the board reads/writes files; git handles history.
- A general project-management tool — it is scoped to the Claude Code dev lifecycle.

## 2. The lifecycle model

SpecManager tracks **features** (initiatives). Each feature traverses five stages; stage *N* unlocks once stage *N−1* is **approved** (the generation gate).

| # | Stage | Documents produced | Generated from |
|---|-------|--------------------|----------------|
| 1 | **PRD** | `press-release.md`, `prd.md` | user intent / notes |
| 2 | **Architecture** | `architecture.md` | approved PRD **+ the existing codebase** |
| 3 | **Plan** | `plan.md` (milestones/phases) **+ task records** (`tasks.md`, `tasks.json`) | approved Architecture + codebase |
| 4 | **Build** | *(no doc — execution)* | approved Plan |
| 5 | **Walkthroughs** | `<slug>.md` per feature | Build complete + actual code |

**Per-document status:** `draft → approved` (no separate "review" state in v0).
**Per-task status:** `todo → in_progress → done`.
**Stale flag:** any document also carries a `stale` badge, independent of status — see §2.2.

### 2.1 Planning vs. building vs. documenting

Stages 1–3 produce **approved planning artifacts** — the **Plan** stage yields both the plan and its task breakdown in one step. **Build** has no document of its own; it's the implementation phase, tracked by its tasks' `todo → in_progress → done`, and is *complete* when every task is `done`. **Walkthroughs** therefore gate on tasks **done**, not on an approved doc — the one stage whose gate is *completion* rather than *approval*.

### 2.2 Living docs — revision & staleness

`approved` is a human *commitment*, not a freeze. To change an approved doc you **reopen** it (→ `draft`); on reopen, every downstream doc that depends on it (via `dependsOn`) is auto-flagged **`stale`** — a *non-blocking* "needs review" badge. You keep working; the badge just signals the basis changed. Reconciling a stale doc (regenerate, or review + re-approve) clears its flag. `core` records **`basedOn`** (the upstream ids + versions a doc was generated from), so staleness is computed precisely rather than guessed.

### 2.3 The board

The board is a **grid**: each **row is a feature**, each **column is a stage** (PRD · Architecture · Plan · Build · Walkthroughs). Every row has one **card per column** — that feature's artifact for that stage — and cards fill in **left to right** as the feature progresses. A card shows its status (`draft`/`approved`), a `stale` badge if flagged, and "generated by agent/human"; the **Build** cell shows a `todo/in_progress/done` progress bar instead of a document. The gate prevents a stage's *Generate* action until the prior stage is approved (or, for Walkthroughs, until tasks are done). See §8.

## 3. Source-of-truth layout (inside the user's project)

```
<project>/
  CLAUDE.md                    # project constitution; holds a managed SpecManager block (§10)
  .claude/
    specs/
      manifest.json            # board index (derived/cacheable from frontmatter)
      features/
        <feature-slug>/
          feature.json         # id, slug, title, currentStage, createdAt
          prd/
            press-release.md
            prd.md
          architecture/
            architecture.md
          plan/
            plan.md
            tasks.md           # human-readable task rollup
            tasks.json         # structured task records (status, links)
          walkthroughs/
            <slug>.md
```

Each **feature** is a self-contained pipeline. The **Plan** stage owns both `plan.md` and the task records. Build has no folder — it's execution, tracked via the feature's `tasks.json`. Cross-cutting/project-level architecture (shared ADRs) can live in a reserved `_project` feature; deferred to a later iteration.

### 3.1 Document format

Every markdown document carries YAML frontmatter; the body is the content the user reads/edits.

```yaml
---
id: arch-001
featureId: feat-checkout   # which feature this belongs to
stage: architecture        # prd | architecture | plan | walkthrough
status: draft              # draft | approved
stale: false               # auto-set true when an upstream doc is reopened
title: System Architecture
dependsOn: [prd-001]       # upstream doc ids (drives the gate)
basedOn: { prd-001: 4 }    # upstream id → version this was generated from
generatedBy: agent         # agent | human
version: 3                 # bumped on each write
createdAt: 2026-05-24T10:00:00Z
updatedAt: 2026-05-24T11:30:00Z
---

# System Architecture
...markdown body...
```

### 3.2 Task record (`tasks.json`)

```json
{
  "tasks": [
    {
      "id": "task-014",
      "featureId": "feat-checkout",
      "title": "Implement MCP write_document tool",
      "status": "in_progress",
      "stageRef": "plan-001#phase-2",
      "dependsOn": ["task-013"],
      "artifacts": {
        "commits": ["a1b2c3d"],
        "files": ["server/src/core/documents.ts"],
        "pr": null
      },
      "createdAt": "…", "updatedAt": "…"
    }
  ]
}
```

The `artifacts` block links a task to the code that implemented it (commits/files/PR) — this is what the walkthrough-writer reads to ground its output, and the basis for "Build complete" once every task is `done`.

`manifest.json` is an index built from all frontmatter + `tasks.json`. It is a **cache**: if deleted, the server rebuilds it by scanning `.claude/specs/`. The frontmatter is authoritative.

## 4. System architecture

### 4.1 One shared core, two faces

```
                ┌─────────────────────────────────────────────┐
                │            @specmanager/core (library)          │
                │  read/write/validate · stage state machine   │
                │  gate checks · file watcher · event emitter  │
                └───────────────▲──────────────▲───────────────┘
                                │              │
            MCP (stdio)         │              │      HTTP/REST + WS
        ┌───────────────────────┴──┐      ┌────┴───────────────────────┐
        │      MCP server           │      │      Board server          │
        │  tools: read/write/status │      │  /api/* · /ws · serves the  │
        │  create/gate/tasks        │      │  built React/Vite UI        │
        └───────────────▲──────────┘      └─────────────▲──────────────┘
                        │                                │
                 Claude Code                        Browser (kanban)
              (agents & skills)                   localhost:5173-ish
```

Both servers import the **same `@specmanager/core`** module, so a write is a write — identical validation, identical state transitions, identical events — whether it comes from Claude or from a human dragging a card.

### 4.2 Process model (single-user local)

Recommended default: **the MCP server boots the board server** (or ensures it's running) on startup, so a single `claude` session brings up everything. The `/specmanager:board` skill just opens the browser.

- `.mcp.json` makes Claude Code auto-start the MCP server when the plugin is enabled.
- On start, the MCP server ensures the board HTTP/WS server is listening on a localhost port (binding `127.0.0.1` only).
- The browser loads the prebuilt Vite UI served by the board server and connects to `/ws` for live updates.
- The board server also hosts the **Agent SDK chat orchestrator** (§11), spawning a headless Claude Code session per open document.

Because `@specmanager/core` is a standalone library, you can later split this into an always-on board server (independent of any Claude session) for a future team mode — no rewrite, just a different process topology.

### 4.3 Live sync

- The board server runs a file watcher (e.g. chokidar) on `.claude/specs/**`. Any change → `core` emits an event → broadcast over `/ws` → the board updates without refresh.
- All `core` mutations (from MCP tools or REST) emit the same events, so agent-driven changes appear live too.
- A plugin **`FileChanged` hook** watching `.claude/specs/**` notifies Claude Code when the human edits a doc, so the agent re-reads before its next action.

## 6. MCP tools (the agent's interface)

Exposed by the MCP server, callable by skills and subagents. All mutations route through `@specmanager/core`.

| Tool | Purpose |
|------|---------|
| `specmanager_init` | Scaffold `.claude/specs/` + empty manifest + the `CLAUDE.md` managed block |
| `list_features` / `create_feature` | List features; create one (`title`) → returns `featureId` + slug |
| `list_documents` | List docs, filterable by `featureId` / `stage` / `status` / `stale` |
| `read_document` | Return a document's frontmatter + body by `id` |
| `create_document` | Create a doc in a feature+stage (`featureId`, `stage`, `title`) → `id` |
| `write_document` | Replace body/frontmatter; bumps `version`, emits event |
| `set_status` | `draft ⇄ approved`; **reopening** (`approved→draft`) flags dependents `stale` |
| `check_gate` | For a feature+stage, report whether the prerequisite is met (prior stage `approved`, or — for Walkthroughs — all tasks `done`) |
| `list_stale` | List docs currently flagged `stale`, with the upstream change that caused it |
| `list_tasks` / `create_task` / `update_task` | Task CRUD + status + `artifacts` links |
| `link_documents` | Record a `dependsOn` edge (and `basedOn` version) |
| `board_url` | Return the localhost URL (for `/specmanager:board`) |
| `sync_claude_md` | Rewrite the managed block in the project `CLAUDE.md` (status table + pointers; refresh conventions digest) |

**Gate enforcement** lives in `core`, not in the prompt: a generator skill calls `check_gate` and refuses if the prerequisite isn't met. This keeps the lifecycle honest even if the model is told otherwise. **Staleness** is also computed in `core`: on any `approved→draft` transition (or write to an approved doc), it walks the `dependsOn` graph forward and sets `stale: true` on dependents — non-blocking, cleared when they're reconciled.

## 7. REST + WS API (the UI's interface)

A thin mirror of `core`, served by the board server on localhost.

```
GET    /api/board                     → grid: features (rows) × stages (columns)
GET    /api/features                   → features
POST   /api/features                   → { title } → new feature (+ slug)
GET    /api/features/:id               → a feature's docs across all stages
GET    /api/documents/:id             → frontmatter + body
POST   /api/documents                 → { featureId, stage, title } → new doc
PUT    /api/documents/:id             → update body/frontmatter
POST   /api/documents/:id/status      → { status }  (draft ⇄ approved; reopen flags dependents)
GET    /api/stale                      → docs currently flagged stale
GET    /api/tasks?featureId=          → tasks (filterable by feature)
POST   /api/tasks  /  PUT /api/tasks/:id
WS     /ws                            → server→client events:
                                         feature.created | document.created | document.updated
                                         status.changed   | stale.flagged | task.updated
                                         file.changed
                                         chat.delta | chat.tool | chat.done | chat.error
                                       → client→server (chat, §11):
                                         chat.send { docId, text } | chat.cancel | chat.mode
```

## 8. The React / Vite UI

**Board = grid.** Rows are **features**; columns are the five **stages** (PRD · Architecture · Plan · Build · Walkthroughs). Each row has one **card per column** — that feature's artifact for that stage — and cards fill **left to right** as the feature advances. A "+ Feature" action adds a new row.

- Each cell shows the doc's title, status (`draft`/`approved`), a **`stale`** badge if flagged, and "generated by agent/human". The **Build** cell shows a `todo / in_progress / done` progress bar instead of a document.
- Empty/locked cells (the prior stage isn't approved yet) render as a dimmed **Generate** affordance.
- **Open a cell** → split view: rendered markdown + editor.

**Key interactions**

- **Open a card** → split view: rendered markdown + editor (CodeMirror or TipTap/Milkdown). Save → `PUT /api/documents/:id`.
- **AI chat panel** (§11) → sits beside the editor; interviews you on a blank doc, co-writes once there's content. Its edits stream into the same editor live.
- **Status control** → `draft → approved`. Approving a stage lights up the next stage's **Generate**. Approved docs are read-only until you hit **Reopen** (→ `draft`), which flags dependents `stale`.
- **Stale badge** → click to see what upstream change caused it and jump to reconcile; non-blocking.
- **Generate button** (per stage) → shows the exact slash command (e.g. `/specmanager:architecture`) or, if wired, triggers it. Walkthroughs' Generate unlocks when all tasks are `done`.
- **Live updates** via `/ws`: agent edits, status/stale changes, and file changes animate in without refresh.

**Suggested libraries:** React 18, Vite, `@dnd-kit` (drag/reorder), a markdown editor (CodeMirror 6 or TipTap), `zustand` or React Query for state, and a small WS client. *(When you build the UI, apply the `frontend-design` skill for styling.)*

## 9. Agent workflow (skills & subagents)

Each stage is a **skill** that delegates to a focused **subagent**. Plugin subagents automatically have access to the plugin's MCP tools (note: plugin agents cannot declare their own `mcpServers`/`hooks`/`permissionMode` — they inherit the session's). Skills take a `featureId` so they operate on one feature's pipeline.

Typical `/specmanager:architecture <feature>` flow:

1. Skill calls `check_gate(featureId, "architecture")`; aborts clearly if the feature's PRD isn't approved.
2. Invokes the **architect** subagent.
3. Architect reads the approved PRD via `read_document` **and inspects the existing codebase** (it has the repo, like any Claude Code session), drafts the architecture, calls `create_document`/`write_document`, records `basedOn`, leaves status `draft`.
4. The feature's Architecture card appears as a draft; you edit and set **approved**.
5. That unlocks `/specmanager:plan`, and so on.

**Plan = plan + tasks.** `/specmanager:plan <feature>` runs the **planner** subagent, which produces `plan.md` *and* the task records (`tasks.json` + rollup) in one step; approving the Plan stage commits both.

**Build & Walkthroughs.** There's no Build generator — implementation is ordinary Claude Code work against the approved Plan's tasks, with task status and `artifacts` updated via `update_task` as code lands. `walkthrough-writer` gates on **all tasks `done`**, then reads `tasks.json` + the linked code to produce per-feature walkthroughs. **Reconciling stale docs** is the same skills run again on a flagged doc, after which it's re-approved and its flag clears.

Example subagent frontmatter:

```markdown
---
name: architect
description: Drafts system architecture from an approved PRD and the existing codebase. Invoke for /specmanager:architecture.
model: sonnet
effort: high
maxTurns: 30
---
You are a pragmatic software architect. Read the approved PRD via the specmanager
read_document tool AND inspect the existing repo (Read/Grep) so the design fits
what's already there; then produce a clear architecture.md … (system prompt continues)
```

## 10. Leveraging the project `CLAUDE.md`

The project's root `CLAUDE.md` is the agent's constitution — read **every session, before any skill runs**. SpecManager uses it as an always-on summary of lifecycle state, so Claude is oriented even outside a `/specmanager:*` command. (Note: a `CLAUDE.md` shipped *inside the plugin* is **not** loaded as project context — plugins contribute context via skills, agents, and hooks. SpecManager therefore writes into the *project's* `CLAUDE.md`, never its own.)

### 10.1 The managed block

SpecManager owns one delimited region; everything outside the markers is yours and is never touched.

```markdown
<!-- specmanager:start -->
## Project lifecycle (managed by SpecManager — do not edit by hand)

Specs live in `.claude/specs/features/`. Read the approved doc for a feature's stage before implementing it.

| Feature | Current stage | Notes |
|---------|---------------|-------|
| Checkout corridor | Build (4/9 tasks done) | Plan ⚠️ stale — PRD was reopened |
| KYC onboarding | Architecture (approved) | — |
| Referral program | PRD (draft) | — |

**Rules:** don't start a feature's tasks until its Plan is approved; treat ⚠️ stale docs as needing reconciliation.

_Last synced: 2026-05-24T11:30:00Z_
<!-- specmanager:end -->
```

### 10.2 What goes in (and what doesn't)

- **Status block** — deterministic, written by `@specmanager/core`: the stage table, the path to `.claude/specs/`, and the gate rule. Always accurate, cheap to keep loaded.
- **Conventions digest** — agent-written, refreshed when **Architecture** is approved: stack decisions, build/test commands, and directory conventions distilled from `architecture.md`, so the constitution stays consistent with the approved design.
- **Pointers, not bodies** — the block links to specs; full content is read on demand via `read_document`, keeping the always-loaded token cost small.

Bulkier, stage-specific guidance can live in `.claude/rules/*.md` instead, which load on the same mechanism without bloating the main file.

### 10.3 How it stays in sync

- **`sync_claude_md`** regenerates *only* the managed block, idempotently, between the markers — hand-written content and git history outside the block are untouched.
- It runs on **meaningful events only**: when a doc is `approved`, reopened, or flagged `stale`, when a feature is created, and on `/specmanager:init`. No churn on every keystroke. A manual `/specmanager:sync` is available too.
- An **`InstructionsLoaded` hook** (matcher `CLAUDE.md`) fires when the file loads and can flag a stale block ("specs changed since last sync — run `/specmanager:sync`").

### 10.4 Conflict & noise discipline

- Stay strictly inside the markers; if they're missing, recreate them at the top rather than guessing where edits belong.
- Keep the block small and stable so diffs stay minimal and reviewable.
- Commit `CLAUDE.md` — the block reflects the last approved state, so a fresh clone is instantly oriented.

## 11. In-UI AI chat (Agent SDK)

Each spec card opens a chat panel beside the editor, powered by a headless Claude Code session via the **Claude Agent SDK** — the same engine, tools, and context management as Claude Code. The board server spawns and manages the session, with the SpecManager MCP server attached so the chat can read and write the very document you're editing.

### 11.1 How it connects

- The board server uses `@anthropic-ai/claude-agent-sdk` (TypeScript) to start a session scoped to `${CLAUDE_PROJECT_DIR}`.
- **Auth:** reuses your Claude subscription login — the same credentials Claude Code uses, no API key. (Note the separate monthly Agent SDK credit from 2026-06-15.)
- The session is configured with: the **SpecManager MCP server** (so `read_document` / `write_document` / `check_gate` are available), the stage's **subagent persona** as system prompt (PRD-writer for a PRD card, architect for an architecture card, …), and an allow-list limited to the SpecManager tools + `Read`, so the chat grounds in the repo but won't run arbitrary edits or Bash unless you opt in.
- **One session per open document**, keyed by doc id; reused across turns for context; torn down when the card closes or after an idle timeout.

### 11.2 Adaptive behavior

- **Blank doc → interview.** With an empty body, the session opens in interview mode: focused questions one topic at a time (working-backwards prompts for a PRD; constraints/NFRs for architecture; etc.), then it drafts the document.
- **Has content → co-writer.** Once the doc has substance, it switches to co-writing: you ask for changes ("tighten the goals", "add a risks section") and it edits in place.
- Mode is derived from the doc's state on session start and can be toggled manually (`chat.mode`).

### 11.3 How edits land

The chat writes through the **same `write_document` path** as everything else — one source of truth, automatic `version` bump, a `document.updated` WS event, live editor update. Hardened so it never clobbers your work:

- **Optimistic concurrency:** every AI write carries the base `version` it read; if you've edited since, the write is rejected and the session re-reads and reconciles (the `version` field from §3.1).
- **Visible & reversible:** AI-written ranges are highlighted in the editor; each change is one WS event you can undo, and git holds history.
- **The gate is the checkpoint, not each keystroke:** the doc stays `draft` until you set `approved`, so you review holistically at stage approval rather than approving every edit — keeping co-writing fluid. If you later want stricter control, a diff-accept step is purely an editor concern, not an architecture change.

### 11.4 Chat transport (WS)

Chat rides the existing `/ws` channel:

- **client → server:** `chat.send { docId, text }`, `chat.cancel { docId }`, `chat.mode { docId, mode }`
- **server → client:** `chat.delta { docId, token }` (streamed assistant text), `chat.tool { docId, name, status }` (e.g. `write_document` running), `chat.done`, `chat.error`

Document edits still arrive via the normal `document.updated` event, so the editor updates through one path whether a change came from the chat, a slash command, or a file edit.

### 11.5 Relationship to the slash-command flow

The slash-command generators (§9) and the in-UI chat are two front-ends to the **same engine and tools**. Slash commands suit whole-stage generation inside a terminal session; the chat suits iterative authoring inside the board. Both write through `write_document` and respect the gate, so they're interchangeable and stay consistent.

## 12. Tech stack

- **Runtime:** Node 20+, TypeScript.
- **MCP:** `@modelcontextprotocol/sdk` (stdio transport).
- **In-UI AI:** `@anthropic-ai/claude-agent-sdk` (TypeScript), subscription auth, streamed over WS.
- **Board server:** Fastify (or Express) + `ws`; `chokidar` for file watching.
- **Docs:** `gray-matter` (frontmatter), `zod` (schema validation), `marked`/`markdown-it` for rendering server-side if needed.
- **UI:** React 18 + Vite, `@dnd-kit`, CodeMirror 6 or TipTap, React Query/zustand.

## 13. Security & local considerations

- Bind board server to `127.0.0.1` only; no auth in single-user local mode.
- No secrets required; nothing leaves the machine.
- `.claude/specs/` is git-tracked; `manifest.json` is a cache (you may `.gitignore` it or commit it — committing keeps the board instant on clone).
- Resolve the project root from `${CLAUDE_PROJECT_DIR}`, never from cwd.

## 14. Phased implementation plan

**Principle:** every phase ends with a plugin you can install into a real repo (`claude plugin install ./specmanager --scope local`, or via a local marketplace) and exercise end-to-end. Each phase builds on the last; nothing is thrown away. Each lists what it **adds** and a concrete **install & test** acceptance check.

**Phase 1 — Headless core (no UI yet).**
*Adds:* `plugin.json` + `.mcp.json`; `@specmanager/core` (feature + doc schema, `draft⇄approved` machine, `dependsOn`/`basedOn` graph, staleness propagation); the MCP server exposing the §6 tools (features, doc CRUD, `set_status`/reopen, `check_gate`, `list_stale`, tasks); the `/specmanager:init` and `/specmanager:feature` skills.
*Install & test:* install into a scratch repo → `/specmanager:init` scaffolds `.claude/specs/` + the `CLAUDE.md` block → create a feature, create a PRD doc, approve it, then reopen it and confirm a dependent is flagged `stale` — all from the Claude Code session, files visible in git. **A useful plugin already, no UI required.**

**Phase 2 — Read-only board.**
*Adds:* the board server (REST `/api/board` · `/api/features` · `/api/documents/:id`, WS, chokidar watcher) hosted in the MCP server process; a prebuilt React **grid board** (rows = features × five stage columns, read-only cells); `/specmanager:board` to open it.
*Install & test:* `/specmanager:board` opens localhost; the features/docs from Phase 1 render in the right stages; editing a spec file on disk updates the board live.

**Phase 3 — Edit, approve & stale in the UI.**
*Adds:* markdown viewer/editor (save via `PUT`), `draft⇄approved` + **Reopen** controls, stale badges with "what changed" + jump-to-reconcile, gate-driven **Generate** buttons (surfacing the slash command).
*Install & test:* edit a PRD in the UI and save; approve it; reopen it and watch the Architecture card gain a stale badge; confirm Generate stays disabled until the gate is met.

**Phase 4 — Agent stage generators.**
*Adds:* the stage skills + subagents (`prd-writer`, `architect`, `planner`, `walkthrough-writer`), feature-scoped and **brownfield-aware** (read the repo); the **planner produces `plan.md` + task records in one step**; gate enforcement via `check_gate`; `sync_claude_md` on approve/reopen/stale; the `FileChanged` + `InstructionsLoaded` hooks.
*Install & test:* in a real repo, `/specmanager:prd <feature>` drafts a PRD → approve → `/specmanager:architecture <feature>` drafts from the PRD **and** the existing code → `/specmanager:plan <feature>` emits a plan with its tasks; the `CLAUDE.md` block updates automatically.

**Phase 5 — Build execution & walkthroughs.**
*Adds:* `create_task`/`update_task` with `artifacts` links (commits/files/PR), the **Build** column with a progress bar, and walkthrough generation gated on tasks `done`. (Task *records* already exist from the Plan stage in Phase 4; this phase makes execution and progress first-class.)
*Install & test:* take an approved Plan's tasks; mark them `in_progress`/`done` with commit/file links and watch Build progress; once all are `done`, generate a walkthrough grounded in the linked code.

**Phase 6 — In-UI AI chat.**
*Adds:* the Agent SDK orchestrator (`agent-chat.ts`, subscription auth), one session per open document with the SpecManager MCP server + stage persona attached, adaptive interview/co-writer modes, the chat WS protocol (§11.4), and live edits via `write_document` with version-checked concurrency.
*Install & test:* open a blank PRD → the chat interviews you and drafts it live; open an existing doc → co-write ("add a risks section"); make a manual edit mid-stream and confirm the version check prevents clobbering.

## 15. Open questions / future

- **Staleness depth:** does `stale` propagate one hop or transitively (reopening PRD → Architecture stale → Tasks stale)? Suggest transitive but collapse the badge to "needs review (upstream changed)" to avoid alarm fatigue.
- **Cross-feature/shared architecture:** project-wide ADRs and dependencies *between* features (v0 treats features as independent). The reserved `_project` feature is the likely home.
- **Generate from the UI:** the chat (§11) covers iterative authoring; whole-stage one-click generation from the board is still a possible add.
- **Team mode:** split the always-on board server out (it already shares `@specmanager/core`), add auth and per-user presence.
- **Concurrent authoring:** chat and a slash-command run can both target one doc; the `version` check (§11.3) rejects the loser — fine for single-user, revisit for teams.
- **History:** rely on git; optionally add a "diff vs last approved" view in the board.
- **Chat cost & session lifecycle:** the Agent SDK draws from the monthly Agent SDK credit (from 2026-06-15); tear sessions down on idle and consider a per-document turn budget, surfacing remaining credit if the SDK exposes it.
