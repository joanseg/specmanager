# Phase 7.C — Test walkthrough

End-to-end test of the **board UI for phased builds**: the Build column groups tasks under collapsible phase headers with per-phase progress bars; the Walkthrough column shows one card per phase plus a feature-level roll-up; and `/specmanager-walkthrough <feature> final` writes `walkthroughs/<slug>/feature.md` that links each phase walkthrough — gated on every phase walkthrough being approved.

> Exit criterion (from `docs/phase-7-execute-and-phased-plans.md`, Phase 7.C):
> Open the board; Build cell groups tasks under collapsible phase headers with per-phase progress bars; Walkthrough column shows one card per phase (states: locked / ready / drafted / approved) plus a final-walkthrough card that stays locked until every phase walkthrough is `approved`; an "Execute next phase" affordance triggers `/specmanager-execute <feature> next` (slash-command copy pattern); once every phase walkthrough is approved, `/specmanager-walkthrough <feature> final` writes `walkthroughs/<slug>/feature.md`.

Assumes Phases 1–7.B already pass.

## 0. Prerequisites

- A repo where SpecManager Phase 7.B already works.
- No new env vars or external services needed.

## 3. Phase 7.C exit checks

### 3.1 Build column groups tasks under phase headers

With a feature whose plan has been generated by the (Phase 7.A) planner — i.e. has multiple phases — open the Build panel by clicking its build cell.

Expected:
- Tasks are no longer a flat list. Each phase appears as its own collapsible section: `▾ Phase <name>  <done>/<total> done` with a per-phase progress bar.
- Click the caret to collapse/expand the phase's task list.
- A faint `/specmanager-execute <feature> <phase>` link in the header copies the slash command on click.
- The legacy single-phase / pre-7.A case (one synthetic `default` phase) still renders as a flat list — no phase chrome.

### 3.2 Build cell on the board summarises phases

The Build cell on the main board row now shows `<done>/<total> done · phases <doneN>/<totalN>` for multi-phase features. Below the progress bar, a `▶ /specmanager-execute <feature> next` link copies the slash command for the next non-done phase.

### 3.3 Walkthrough column shows N phase cards

The Walkthrough column on the main board row now contains one **sub-card per phase**, in order, plus a final **★ Feature roll-up** card.

States per phase card:
- **Locked** (dashed border, dim): the phase has open tasks. Sub-text shows `<done>/<total> tasks done`.
- **Ready** (accent border): the phase is done but the walkthrough doesn't exist yet. Shows a copy-to-clipboard `/specmanager-walkthrough <feature> <phase>` slash command.
- **Drafted** (regular card with `draft` badge): walkthrough exists, not yet approved.
- **Approved** (regular card with `approved` badge): walkthrough is approved.

### 3.4 Final-walkthrough card stays locked

The **★ Feature roll-up** card sits below the phase cards.

- Hover the card. The tooltip says either `awaiting approval: phase A, phase B` (when some phase walkthroughs aren't approved) or `all phase walkthroughs approved — ready to draft` (when ready).
- It stays locked (dashed border, dim) until **every** phase walkthrough is `approved`.
- Once unlocked, it shows the `/specmanager-walkthrough <feature> final` copy link.

### 3.5 Per-phase walkthrough gate (already 7.B, re-verify)

```
check_gate featureId=<id> stage="walkthrough" phase="A"
```

Expected: `ok: false` with reason naming undone tasks while A has work; `ok: true` once all of A's tasks are `done`.

### 3.6 Final-walkthrough gate (new in 7.C)

With Phase A walkthrough still `draft`:

```
check_gate featureId=<id> stage="walkthrough" phase="final"
```

Expected: `ok: false` with `reason: "phase walkthroughs not approved: A, B"` (or just `B` if A is approved).

After approving every phase walkthrough:

```
check_gate featureId=<id> stage="walkthrough" phase="final"
```

Expected: `ok: true`.

### 3.7 Final walkthrough lands at `feature.md`

After every phase walkthrough is approved:

```
/specmanager-walkthrough <feature> final
```

Expected:
- Gate opens, `walkthrough-writer` subagent runs in **final mode**.
- New doc lands at `.claude/specs/features/<slug>/walkthroughs/feature.md`.
- Frontmatter has `phase: final`.
- Body links each phase walkthrough (`walkthroughs/<slug>/phase-<name>.md`) and revisits PRD success metrics.
- It does NOT contain a new code tour — phase walkthroughs own that.

```bash
ls .claude/specs/features/<slug>/walkthroughs/
# phase-a.md  phase-b.md  feature.md
```

### 3.8 Live updates

With the board open in a browser, in another terminal mark a task done with artifacts via MCP or REST:

```bash
curl -X PATCH http://127.0.0.1:4317/api/features/<id>/tasks/<task> \
  -H 'content-type: application/json' \
  -d '{"status":"done","artifacts":{"files":["src/x.ts"]}}'
```

Expected: the board's per-phase progress bar advances within ~100ms; if the phase becomes complete, its walkthrough sub-card transitions from `locked` to `ready` with the copy-slash affordance.

### 3.9 Refuse `final` early

```
/specmanager-walkthrough <feature> final
```

…with at least one phase walkthrough still `draft`.

Expected: refusal with the gate reason. The card on the board stays locked.

## 4. Out of scope for 7.C

- Auto-approval of walkthroughs (still user-driven).
- Cross-feature roll-ups.
- Parallel phase execution (phases stay sequential — the testable boundary).
