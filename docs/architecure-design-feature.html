Design feature — Architecture

Introduce a first-class Design stage between Architecture and Plan, plus a repo-level ./docs/DESIGN.md design-system spec that /specmanager-init creates and that re-syncs whenever a feature ships.

1. Summary

SpecManager today walks PRD → Architecture → Plan → Build → Walkthroughs. This feature inserts Design as an optional stage between Architecture and Plan. It ships in three coordinated pieces:

  1. A new design stage in the core schema, with its own column on the board, gate logic that lets the Plan stage proceed whether or not a Design doc exists, and a slash command /specmanager-design driven by a new designer subagent that produces an HTML design brief (screenshots + descriptions).
  2. A repo-level ./docs/DESIGN.md living outside .claude/specs/ — the project's design system spec (tokens, components, layouts) following the Google Stitch DESIGN.md spec. It is generated/refreshed by /specmanager-init from the existing UI code, and auto-re-syncs after every feature's last phase completes.
  3. UI wiring on the kanban board: a new Design column rendered between Architecture and Plan, with an "optional" affordance so users can clearly skip it.

The architect / planner / builder subagents become design-aware: when a Design doc exists for a feature they read it and ground task descriptions in it; when none exists they proceed exactly as today.

Core (server) — add (new)

    • scanUiSources(root) — walk the project's UI directories (default heuristics: src/, app/, ui/, web/, frontend/, plus any directory containing *.tsx/*.jsx/*.vue/*.svelte) and surface a digest: token files (CSS vars, Tailwind config, design-tokens.json), component manifests, layout shells.

On init mode, create or merge; on refresh mode (after a feature ships), regenerate only the auto-managed sections between two HTML-style markers <!-- specmanager:design:start --> / <!-- specmanager:design:end -->, preserving anything outside (same pattern as claude-md.ts).
  • plugins/specmanager/server/src/core/paths.ts — add designMdPath(root) returning <root>/docs/DESIGN.md.
  • plugins/specmanager/server/src/core/init.ts — extend initProject to also call syncDesignMd({ mode: "init" }); widen InitResult with designMd: string and createdDesignMd: boolean.
  • plugins/specmanager/server/src/core/events.ts — add a feature.shipped event emitted when the last phase of a feature's walkthrough is approved. The MCP server's existing auto-sync wiring (Phase 4) subscribes and triggers syncDesignMd({ mode: "refresh" }) with a debounce.

5. Sequence & flow

5.1 — /specmanager-init on a fresh repo

  1. User runs /specmanager-init in a project.
  2. Slash command calls specmanager_init MCP tool.
  3. initProject runs in order: ensureSpecsRoot → migrateWalkthroughs → syncDesignMd({mode:"init"}) → writeManifest → syncClaudeMd.
  4. syncDesignMd calls scanUiSources(root), infers tokens/components/layouts from the existing UI, renders DESIGN.md content between managed markers, and writes ./docs/DESIGN.md (creating docs/ if absent).
  5. Result reported to the user includes designMd and createdDesignMd.

5.2 — /specmanager-design for a feature

  1. User runs /specmanager-design feat-checkout. Optionally pastes screenshot paths and a one-line brief in the conversation first.
  2. Slash command resolves the feature, calls check_gate({featureId, stage: "design"}). Gate passes if PRD is approved.
  3. Slash command spawns the designer subagent via Task with the PRD id, the Architecture id (if approved), the ./docs/DESIGN.md path, and any attachment paths.
  4. The subagent reads the upstream docs and DESIGN.md, generates an HTML body that includes embedded screenshot data URIs (read via Read tool, base64-encoded), and calls create_design_brief.
  5. create_design_brief persists the doc via the existing createDocument path. Frontmatter has stage: "design", status: "draft", basedOn: { <prdId>: prdVersion, <archId>: archVersion }.
  6. Existing Phase 4 auto-sync emits document.created → CLAUDE.md refreshes.
  7. User reviews in the doc panel (HTML-aware), iterates with the chat panel from Phase 6, and approves.

5.3 — Planner / builder ground in the design when present

  1. /specmanager-plan feat-checkout resolves the feature, calls check_gate({stage:"plan"}) — passes only when both architecture is approved AND (no design doc OR design approved).
  2. Slash command discovers the design doc via list_documents({featureId, stage:"design"}); if present, passes its id into the planner's Task prompt as additional grounding.
  3. Planner reads PRD + Architecture + Design (if any) + repo conventions and produces plan.md + tasks. Tasks that have an obvious design counterpart get a stageRef citing the design doc id.
  4. Builder subagent for each phase similarly reads the design doc if present and references it when wiring up components.

5.4 — Feature completion → DESIGN.md refresh

  1. User approves the final walkthrough for a feature.
  2. setStatus sees doc.frontmatter.stage === "walkthrough" && doc.frontmatter.phase === "final" and emits feature.shipped.
  3. The board server's event listener (mirroring the Phase 4 auto-sync wiring) debounces 250ms then calls syncDesignMd({mode:"refresh"}).
  4. syncDesignMd re-scans UI sources, regenerates only the managed region of DESIGN.md, writes the file, and emits design.synced. The board pulses to confirm.

6. Failure & edge cases

CaseBehaviour
No UI directory in the reposcanUiSources returns an empty digest; DESIGN.md is created with section stubs and a note explaining no UI was found yet. /specmanager-init still succeeds.
DESIGN.md already exists, hand-edited, no managed markersTreat the existing content as preamble; insert managed markers + auto-content at the bottom. Never overwrite hand-written content above the markers.
User attaches a 50MB screenshotcreate_design_brief rejects attachments > 2MB with a clear error. Recommended path: commit the asset to the repo and reference it relatively in the body instead of inlining as data URI.
Two /specmanager-design runs land back-to-backcreateDocument's existing "document file already exists" guard fires; the second invocation gets a clean error pointing at the existing brief. Same behaviour as Phase 4 stage subagents.
Plan started without a Design doc, then user runs /specmanager-design laterAllowed. The design doc lands as draft; Plan is unaffected (it was generated before the design existed). On the user's next Plan iteration, the design doc shows up as upstream context — Phase 3's stale-propagation mechanism marks Plan stale because of the new basedOn edge if the user explicitly links them via link_documents.
Designer subagent produces malformed HTMLBody is stored verbatim — markdown ecosystem tolerance is irrelevant since the panel uses <iframe srcdoc>. The iframe sandbox (sandbox="allow-same-origin" only, no scripts) contains any breakage.
Feature ships, DESIGN.md refresh fails (disk full, permissions)The board surfaces design.synced with a failure flag; the failure is also logged to MCP server stderr. The feature ships regardless — DESIGN.md refresh is best-effort.
Design doc reopened (approved → draft) while Plan is in flightExisting Phase 3 stale-propagation kicks in: Plan is flagged stale via the dependsOn graph if it was linked. Visible in the board UI.

7. Conventions used

  • Gate enforcement in core, not prompts — the new design gate edge lives in dependencies.ts::PRIOR_STAGE and the OR'd "design approved or absent" check is in checkGate. Slash commands call check_gate; refusal text is derived from the core's reason.
  • Managed-block pattern — DESIGN.md uses the same <!-- specmanager:start --> / <!-- specmanager:end --> idiom as claude-md.ts, just with the marker namespace specmanager:design. Anything outside the markers is the user's territory.
  • One MCP process boots the board — no new processes. The DESIGN.md refresh listener piggybacks on the existing event bus (Phase 4 wiring) in the same Node process.
  • Auto-sync via events, not direct calls — both CLAUDE.md and DESIGN.md auto-sync are driven by the typed event bus. Mutation paths (MCP tools, REST endpoints) stay agnostic.
  • HTML brief is a first-class doc, not an attachment — same gray-matter frontmatter, same versioning, same optimistic concurrency. Only the body format differs.
  • Frontmatter remains authoritative; manifest is rebuildable — design docs work the same way.
  • Subagent + slash-command + MCP tool triplet — matches the Phase 4 pattern for every stage generator.

8. Open questions & risks

Q1 — Google Stitch DESIGN.md spec details. The architecture assumes the Stitch spec's section structure (Overview, Tokens, Typography, Color, Components, Layouts, Motion, A11y). If the spec is materially different, the renderDesignMd function changes — but the surrounding wiring (init hook, refresh trigger, managed markers, MCP tool) doesn't. Resolve by pinning a snapshot of the spec into docs/references/stitch-design-md.md before implementation starts.
Q2 — Screenshot storage strategy. Inline data: URIs are simplest (briefs are self-contained, no asset directory to manage), but they bloat the doc file. Alternative: commit screenshots to .claude/specs/features/<slug>/design/assets/ and reference them relatively. The architecture currently picks inline-with-a-2MB-cap; the planner phase should reconsider if any real brief blows past it.
R1 — Gate complexity creep. Plan's gate becomes a compound expression (architecture AND (no design OR design approved)). This is the first compound gate in checkGate; the test matrix grows. Mitigated by adding ≥4 selftest cases covering all combinations: (no design + draft arch), (no design + approved arch), (draft design + approved arch), (approved design + approved arch).
R2 — DESIGN.md refresh might surprise users. A user who hand-edited DESIGN.md preamble could be alarmed if they didn't realise the auto-managed region exists. Mitigation: /specmanager-init output explicitly lists DESIGN.md as created/touched, the markers are loud, and the README's Phase doc explains the contract.
R3 — HTML-in-frontmatter parsing. gray-matter uses --- as the frontmatter delimiter. If a designer's HTML body contains --- at column 0, parsing breaks. Mitigation: create_design_brief wraps any leading --- in the body in an HTML comment before write. Cheap, defensive.
R4 — Build chain divergence. The architect / planner / builder agents all need the "if design exists, read it" addendum. If one agent forgets, design decisions don't propagate. Mitigation: one shared prompt fragment docs/agent-snippets/design-grounding.md referenced by all three agent prompts, plus an end-to-end test that grep's the generated plan for a known design-doc reference.
